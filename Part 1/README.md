# Part 1 — Data Acquisition, Cleaning, and Exploratory Analysis

## 1. Dataset description and justification

**File:** `employee_raw.csv` (668 rows × 11 columns before cleaning)

This is a synthetic HR-analytics dataset (employees, departments, salary,
performance, satisfaction). It was generated by `generate_raw_data.py` rather
than downloaded from a public repository, **because the environment used to
produce this submission has no internet access**. The generator was written
to reproduce the exact kind of problems a real raw export would have, on
purpose, so every cleaning step in `analysis.py` has a genuine issue to fix
rather than a contrived one:

- **Numeric target column:** `monthly_salary`
- **Categorical columns:** `department`, `education_level`, `region`, `remote_work`
- **Other numeric columns:** `age`, `years_experience`, `performance_score`, `satisfaction_score`, `bonus_pct`
- **Rows:** 668 (>500 required), **Columns:** 11 (>5 required)

Columns:

| Column | Type | Description |
|---|---|---|
| employee_id | int | Unique identifier |
| age | float | Employee age (has outliers, has nulls) |
| department | categorical | Sales / Engineering / Marketing / Support / Finance / HR |
| education_level | categorical | High School / Bachelors / Masters / PhD |
| region | categorical | North / South / East / West |
| remote_work | categorical | Yes / No |
| years_experience | float | Years of professional experience (correlated with age & salary) |
| monthly_salary | **stored as text** (e.g. `"$12,147.21"`) | Monthly salary — the numeric target |
| performance_score | float | 1–10 performance rating (has nulls) |
| satisfaction_score | float | 1–100 satisfaction rating (has nulls) |
| bonus_pct | float | Bonus percentage (heavy nulls, right-skewed) |

If you want to substitute a real public dataset instead, drop any CSV with
≥500 rows / ≥5 columns (numeric + categorical) in as `employee_raw.csv` with
matching column names, or edit the column names at the top of `analysis.py`.

**Reproducing the raw data:** `python3 generate_raw_data.py` (seeded, so it
is deterministic).

## 2. How to run

```bash
pip install pandas numpy matplotlib seaborn scipy
python3 generate_raw_data.py   # creates employee_raw.csv (already included)
python3 analysis.py            # runs all cleaning/EDA tasks, writes plots + cleaned_data.csv
```

Outputs:
- `cleaned_data.csv` — final cleaned dataset (650 rows × 11 columns)
- `plots/01_line_monthly_salary.png` … `plots/06_correlation_heatmap.png`
- `console_output.txt` — full printed output of the run (null tables, stats, etc.)

## 3. Null value analysis (Task 2)

| Column | Null count | Null % |
|---|---|---|
| age | 33 | 4.94% |
| performance_score | 52 | 7.78% |
| satisfaction_score | 80 | 11.98% |
| bonus_pct | 236 | **35.33%** |
| all others | 0 | 0% |

**`bonus_pct` exceeds the 20% threshold** and was left un-imputed at this
stage per the task instructions (candidate strategies for Part 2: drop the
column, use a missing-indicator flag, or model-based imputation — a decision
deferred to Part 2 since 35% missingness is too high to trust a simple
median fill).

`age`, `performance_score`, and `satisfaction_score` were all below 20% and
were filled with the column **median**, not the mean, because:
1. `age` and `performance_score` both contain a handful of extreme outliers
   (see Task 6) that pull the mean away from the typical employee; the median
   is robust to those extremes.
2. More generally, mean imputation systematically shrinks variance and biases
   the imputed values toward whichever tail the outliers sit on, whereas the
   median reflects the "typical" row regardless of skew — a safer default
   when a column's distribution shape hasn't been fully characterized yet
   (imputation happens in Task 2, before the skewness analysis in Task 5).

## 4. Duplicate detection (Task 3)

18 exact duplicate rows were found and removed (668 → 650 rows). Duplicate
removal changed the null percentage of exactly one column — `bonus_pct`
(35.33% → 34.92%) — because some of the duplicated rows happened to have a
null `bonus_pct` value; every other column's null rate was unaffected since
none of the duplicated rows had nulls in those columns.

## 5. Data type correction (Task 4)

`monthly_salary` was stored as **text** (e.g. `"$12,147.21"`) instead of a
numeric type — a very common raw-export problem when a spreadsheet applies
currency formatting. It was cleaned (`$` and `,` stripped) and converted with
`pd.to_numeric(..., errors='coerce')`.

`department`, `education_level`, `region`, and `remote_work` (all low-
cardinality repeated strings) were converted to `category` dtype.

**Memory usage:** 216,322 bytes → 45,078 bytes (**79.2% reduction**), driven
mostly by the categorical conversion plus the numeric-instead-of-object
salary column.

## 6. Skewness (Task 5)

| Column | Skew |
|---|---|
| **monthly_salary** | **4.90** |
| bonus_pct | 2.03 |
| age | 0.46 |
| performance_score | -0.16 |
| years_experience | 0.15 |
| satisfaction_score | -0.10 |

`monthly_salary` has the highest absolute skewness (4.90), a strong
**positive (right) skew**: most salaries cluster in a moderate range but a
small number of very high earners (executives/outlier records injected on
purpose) stretch the right tail far out. Visually this shows up as a tall
peak near the median with a long thin tail to the right (see the histogram).

**Consequence for mean imputation:** for a positively skewed column, the mean
is pulled well above where most of the data actually sits (mean ≈ 9,044.77
vs. median ≈ 8,601.41 in this dataset), so filling missing salaries with the
mean would systematically overstate a typical employee's pay. The median is
the more representative fill value here.

## 7. Outlier detection — IQR (Task 6)

| Column | Q1 | Q3 | IQR | Lower bound | Upper bound | Outliers |
|---|---|---|---|---|---|---|
| monthly_salary | 7,100.52 | 10,364.12 | 3,263.60 | 2,205.11 | 15,259.52 | **8** |
| age | 32.25 | 45.00 | 12.75 | 13.12 | 64.12 | **3** |

These outliers were **not dropped**. Decision for Part 2: the `monthly_salary`
outliers correspond to genuinely high (but plausible) earners rather than
data-entry errors, so they will be **capped (winsorized) at the upper bound**
before training a linear model in Part 2 (to avoid a few points dominating
the loss), but **retained as-is** for a tree-based model (Part 2's models are
generally robust to outliers in this direction). The `age` outliers (a couple
of values at 80–90) look more like plausible-but-rare senior employees than
errors, so they will simply be retained.

## 8. Visualizations (Task 7)

All six required plots are in `plots/`:

1. **`01_line_monthly_salary.png`** — Salary plotted by row index. No
   meaningful trend across index (the data isn't time-ordered), which
   confirms row order carries no signal — useful to rule out before
   modeling.
2. **`02_bar_mean_salary_by_department.png`** — Mean salary by department.
   Engineering is clearly the highest-paid department, Support the lowest,
   consistent with the department-based salary structure built into the
   synthetic data.
3. **`03_histogram_most_skewed.png`** — Histogram (bins=20) of
   `monthly_salary`. **Shape:** unimodal, right-skewed — a peak around
   $7,000–9,000 with a long right tail extending past $30,000, exactly the
   signature of positive skew described above.
4. **`04_scatter_experience_vs_salary.png`** — `years_experience` vs
   `monthly_salary`. **Interpretation:** a clear positive, moderately strong
   relationship (Pearson r ≈ 0.47) — salary tends to rise with experience,
   though the relationship isn't perfectly linear and scatter increases at
   higher experience levels (more variance among senior employees, likely
   due to department differences).
5. **`05_boxplot_performance_by_department.png`** — `performance_score` by
   `department`. Medians are similar across departments (~7/10), but boxes
   differ somewhat in spread; no department is systematically rated much
   higher or lower, suggesting performance rating is largely independent of
   department.
6. **`06_correlation_heatmap.png`** — Full Pearson correlation matrix.
   **Highest |correlation| pair:** `age` & `years_experience` (r = 0.866).
   This is very unlikely to be purely causal in the sense of one directly
   producing the other — both are almost certainly driven by a **third
   variable: career start age / time since entering the workforce**. Older
   employees have simply had more calendar time available to accumulate
   experience, so the correlation is largely a mechanical consequence of how
   both variables are built from tenure, not evidence that aging itself
   causes more experience (or vice versa).

## 9a. Imputation strategy comparison — mean vs. median (Task 8a)

Two highest-|skew| columns: **`monthly_salary`** and **`bonus_pct`**.

| Column | Mean | Median |
|---|---|---|
| monthly_salary | 9,044.775 | 8,601.41 |
| bonus_pct | 3.560 | 2.48 |

Both columns are **positively skewed** (skew 4.90 and 2.03 respectively), so
in both cases the mean is pulled upward by a minority of extreme high values
— for `monthly_salary` by the injected high-earner outliers, and for
`bonus_pct` by its exponential-shaped tail of employees who received large
bonuses. In both cases the **median** was chosen as the imputation value,
since it better represents where a typical employee actually sits. After
imputing remaining nulls in these two columns with their medians,
`isnull().sum()` confirms **0 nulls remain** in both.

## 9b. Spearman vs. Pearson correlation (Task 8b)

Top 3 column pairs by `|Spearman − Pearson|`:

| Pair | Spearman | Pearson | |Diff| |
|---|---|---|---|
| years_experience & monthly_salary | 0.713 | 0.467 | **0.246** |
| age & monthly_salary | 0.653 | 0.423 | **0.231** |
| age & years_experience | 0.913 | 0.866 | 0.047 |

Interpretation:
- **years_experience & monthly_salary** and **age & monthly_salary**: in both
  cases |Spearman| > |Pearson| by a wide margin, indicating the relationship
  is **monotonic but non-linear** — salary consistently rises with experience
  and age, but not proportionally (consistent with the right-skewed,
  multiplicative way salary was generated: base pay + experience effect × a
  lognormal noise multiplier, which compounds rather than adds linearly).
- **age & years_experience**: |Spearman| and |Pearson| are close (both very
  high, ~0.87–0.91), so this relationship is **approximately linear** — older
  employees have proportionally more experience in a fairly straight-line
  way.
- **For Part 2 feature selection**, Spearman will be the primary guide for
  ranking `years_experience` and `age` as salary predictors, since Pearson
  understates how strongly they actually track salary once the non-linearity
  is accounted for. Pearson remains useful as a secondary check for whether a
  simple linear model term will capture the relationship well on its own, or
  whether a transformation (e.g., log-salary) is needed.

## 9c. Grouped aggregation (Task 8c)

`groupby('department')['monthly_salary'].agg(['mean', 'std', 'count'])`:

| department | mean | std | count |
|---|---|---|---|
| Engineering | 11,937.22 | 3,926.99 | 169 |
| Finance | 8,926.52 | 1,368.73 | 76 |
| HR | 8,209.84 | 2,012.52 | 51 |
| Marketing | 8,144.63 | 1,463.54 | 97 |
| Sales | 7,902.92 | 2,822.13 | 150 |
| Support | 7,375.04 | 3,160.14 | 107 |

- **Highest mean:** Engineering ($11,937.22)
- **Highest standard deviation:** Engineering (3,926.99) — also the
  highest-paid department, so it has both the highest average and the widest
  spread of salaries.
- **Is high within-group std a concern?** Yes — Engineering's large std
  relative to its mean means that knowing only "this employee is in
  Engineering" still leaves substantial uncertainty about their actual
  salary; `department` alone would not reliably predict salary for
  Engineering employees, so Part 2 models should combine `department` with
  other features (experience, performance) rather than relying on it alone.
- **Mean ratio (highest/lowest):** 11,937.22 / 7,375.04 = **1.62**. A ratio
  this large (62% difference between the best- and worst-paid department
  averages) suggests `department` **does carry real predictive signal** for
  salary and is worth keeping as a model feature in Part 2, even though (per
  the point above) it isn't sufficient on its own.
